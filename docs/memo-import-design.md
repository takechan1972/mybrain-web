# メモ取り込み（インポート）機能 ― 設計メモ（OBS43）

> **ステータス：設計のみ・未実装（IMP0）。この OBS43 ではアプリコード・UI・Supabase スキーマ・OAuth スコープ・Google Drive／Obsidian の挙動を一切変更しない。**
> 対象：Obsidian 互換 Markdown ファイルを MyBrain のメモとして取り込む機能の設計（OBS32 次フェーズ D「取り込み（インポート）機能の検討」）。
> 前提：`docs/google-drive-markdown-read-search-design.md`（OBS25〜完了サマリ）／`docs/google-drive-markdown-user-guide.md`（OBS32・4原則）／`docs/markdown-export-qa-checklist.md`（OBS21〜22・エクスポート QA 全9ケース Pass）。
> 実装対象ファイル（将来・IMP1 以降）：`lib/import/`（新規・純ヘルパー）／`components/DesktopSettings.tsx`（既存「⬇ インポート（データ取込）」行の接続）／`lib/storage/supabase-memo-store.ts`（取り込み専用の小さな insert 関数の追加）。

---

## 1. 目的とスコープ

MyBrain はこれまで「書き出し」（Markdown コピー／ダウンロード／ZIP／フォルダ／Google Drive）と「読み取り参照」（Drive 一覧・プレビュー・検索参照・AI 参照）を完成させてきた。残る大きなギャップが**取り込み（インポート）**である：過去にエクスポートした Markdown や、Obsidian 側で書いたメモを、MyBrain のメモとして追加する導線がない。

本設計の範囲（Phase 1）：

- **個別の Markdown ファイル（.md）を選択して、MyBrain（Supabase）へ新しいメモとして追加する。**
- 複数ファイルの同時選択に対応する（1回最大20件）。
- 取り込みは**ユーザー操作起点・プレビュー必須・明示的な確認後のみ保存・insert-only（追加のみ）**。

大原則（既存ドキュメントの4原則を継承）：

- **MyBrain（Supabase）が正本（source of truth）**。取り込みは「Supabase に新しい行を追加する」ことだけを意味する。
- **既存メモは1件も変更・上書き・削除しない**（サイレント上書きの経路を構造的に持たない）。
- **自動保存はしない**。ファイル選択だけでは何も保存されず、確認ボタンを押したときだけ保存される。
- **双方向同期ではない**。取り込みは「ファイル → MyBrain」の一方向・1回きりの手動操作である。

| バッチ結果形の前例 | `GoogleDriveBatchWriteResult`（`lib/google/google-drive-export.ts`） | 成功／失敗／件数サマリの形をそろえる（written / failed / total / successCount / failureCount） |

## 3. Phase 1 対応フォーマット

- **個別の Markdown ファイル（.md）のみ。**
- **複数ファイルの同時選択に対応**（`<input type="file" multiple accept=".md">`。File System Access API は使わないため、対応ブラウザの制限を受けない）。
- **1回の取り込みは最大20件**。21件以上選択された場合は、やさしい案内を出して**取り込みを開始しない**（先頭20件だけ勝手に処理する方式は採らない＝ユーザーの意図しない部分取り込みを避ける）。
- 想定入力は MyBrain 自身のエクスポート（frontmatter 付き）だが、frontmatter 無しの素の Markdown も受け付ける（§6 のフォールバック）。

---

## 4. 後回しフォーマット（Phase 1 ではやらない）

- **ZIP（Markdown 一式）**：次フェーズ（IMP4）。JSZip は依存済みで `loadAsync` により追加ライブラリ不要。単ファイルのパイプラインをそのまま各エントリに適用するだけなので、Phase 1 の実証後に足す。
- **MyBrain JSON**：先送り。**現行 Web 版には JSON エクスポート自体が存在しない**（設定の「⬆ エクスポート」行も準備中）。JSON はエクスポート形式の設計とセットで別 OBS として扱う（画像を往復できる唯一の形式になる見込み）。
- **CSV**：採用しない方向。複数行本文・引用符・文字コード（Excel の Shift_JIS 等）の落とし穴が多く、id／source の由来情報も標準では持てない。旧 Expo 版 CSV の「同じIDは上書き」仕様も現行制約に反する。Markdown／ZIP で用が足りる。

---

## 5. 取り込み候補データモデル（`ImportCandidate` 型案）

プレビュー・確認・保存・結果表示を貫く1件分のデータ。**IMP0 では型の提案のみで、TypeScript ファイルは作成しない。**

```ts
/** 取り込み候補1件の判定状態 */
type ImportCandidateStatus =
  | 'ok'                // 取り込み可能
  | 'duplicate-id'      // 重複（同じメモID）：frontmatter id＋source==="mybrain" が既存メモと一致
  | 'duplicate-content' // 重複の可能性：タイトル＋本文が既存メモと完全一致
  | 'invalid';          // エラー（拡張子・サイズ超過・空・読み取り不能 等）

/** 取り込み候補1件（プレビュー・確認・保存の共通データ） */
interface ImportCandidate {
  /** 選択されたファイル名（例: "買い物メモ.md"） */
  fileName: string;
  /** 判定状態 */
  status: ImportCandidateStatus;
  /** スキップ・エラー時の理由（やさしい日本語。ok のときは null） */
  reason: string | null;

  /** 解析・正規化済みの保存候補データ（invalid のときは null） */
  parsed: {
    /** frontmatter の id（無ければ ''）。重複検知・由来表示のみに使い、保存には使わない */
    sourceId: string;
    /** frontmatter の source（無ければ ''） */
    source: string;
    /** 確定タイトル（frontmatter title → ファイル名 → 保存時の既定「無題」） */
    title: string;
    /** 本文（改変しない） */
    body: string;
    /** 正規化済みタグ（§9） */
    tags: string[];
    /** 保存に使う作成日時（epoch ms。§8 のフォールバック適用済み） */
    createdAt: number;
    /** 保存に使う更新日時（epoch ms。§8 のフォールバック適用済み） */
    updatedAt: number;
    /** created が frontmatter 由来か（false＝取り込み時刻フォールバック） */
    createdFromFrontmatter: boolean;
    /** updated が frontmatter 由来か */
    updatedFromFrontmatter: boolean;
  } | null;

  /**
   * 「重複の可能性」（duplicate-content）の項目だけ、ユーザーが明示的に
   * 「新しいメモとして取り込む」を選べる（既定 false＝スキップ）。
   * duplicate-id と invalid はこのフラグを持たない（常にスキップ）。
   */
  importAsNew: boolean;
}
```

- 判定は**選択直後（プレビュー構築時）に1回**行う。プレビュー表示中に別タブでメモが増えた場合の再判定はしない（許容する限界として §22 に記載）。

## 6. Markdown 解析とファイル名フォールバック

- 解析は既存の `markdownToMemo()` をそのまま使う（**新しいパーサは作らない**）。
- frontmatter 付き：`id / title / tags / created / updated / source` を読む。未知のキーは読まない（§17）。
- frontmatter 無し：全文を本文として扱う（既存フォールバックどおり）。
- **タイトルのフォールバック**（Drive プレビューの「title が無ければファイル名」方式を踏襲）：
  1. frontmatter の `title`（trim 後に非空ならそれ）。
  2. 無ければ**ファイル名から拡張子 `.md`（大文字小文字問わず）を除いたもの**（trim 後に非空ならそれ）。
  3. それも空なら、保存時に `createMemo` 系の既定「**無題**」になる。

---

## 7. 必須・任意フィールド

- **必須フィールドは frontmatter には無い**。ただし解析後に**タイトル・本文の両方が空**のファイルは取り込み対象にならない（`invalid`・理由付きスキップ）。
- 任意：`title`／`tags`／`created`／`updated`／`id`／`source`。すべて欠けていてもよい（フォールバックで補う）。
- `images`（件数メタ）は**無視**する。Markdown は画像データを持たないため、取り込まれたメモの `images` は常に `[]`（§21・画像復元は非目標）。
- 未知の frontmatter キーは一切読み込まない。

---

## 8. 日時の保持とフォールバック（承認済みルール）

**有効な ISO 日時は保持する。**「有効」とは `Date.parse` が成功する ISO 8601 文字列を指す（既存 `isoToMs` の判定と同じ。無効・欠落は 0 になる）。

| 状況 | 保存に使う値 |
|---|---|
| `created` が有効 | **frontmatter の created を保持** |
| `created` が無効・欠落 | **取り込み時刻** |
| `updated` が有効 かつ `updated` ≥ 採用した created | **frontmatter の updated を保持** |
| `updated` が無効・欠落 | **保持した created（それも無ければ取り込み時刻）** |
| `updated` < `created`（逆転） | **updated を無効扱いとし、created を使う** |

- **取り込まれたメモは必ず新しい Supabase メモIDを受け取る**（DB 採番）。frontmatter の `id` を insert に使うことは決してしない。
- **frontmatter の元 `id` の用途は「重複検知（§10）」と「由来（provenance）の表示」のみ。**
- 実現方法（IMP1 で実装・**スキーマ変更なし**）：`memos` テーブルには `created_at` / `updated_at` カラムが既に存在する。現行 `createMemo()` は日時を渡していない（DB 既定＝現在時刻）ため、**取り込み専用の小さな insert 関数**（例：`createMemoWithTimestamps(input, { createdAt, updatedAt })`）を `lib/storage/supabase-memo-store.ts` に追加し、ISO 文字列で明示的に渡す。既存の `createMemo()`・`MemoStore` アダプタインターフェース・通常の保存フローには**一切手を触れない**（既存挙動リスクをゼロにする）。

---

## 9. タグ正規化

- frontmatter の `tags`（フロー配列）を既存パーサで読み、各タグを **trim → 空要素除去 → 同一メモ内の完全一致重複を1つに統合**する。
- それ以外の新しいタグ規則は足さない（小文字化・記号除去・タグ名変換などはしない。既存メモのタグ挙動と同じ扱い）。
- `tags` が無い・空・解析不能のときは `[]`。

---

## 10. 重複検知

判定はすべて**クライアント側**で行う（`listMemos()` は既に全メモをクライアントへ返しており、既存の検索・一覧と同じ方式）。取り込み画面を開いたときに一覧を1回取得し、選択ファイルと突き合わせる。

| 優先順 | 条件 | 判定 | 既定の扱い | ユーザーの選択肢 |
|---|---|---|---|---|
| ① | frontmatter の `id` が既存メモの id と一致 **かつ** `source: "mybrain"` | **重複（同じメモID）** `duplicate-id` | **スキップ** | なし（Phase 1 では常にスキップ） |
| ② | タイトルと本文が既存メモと**完全一致**（①非該当のもの） | **重複の可能性** `duplicate-content` | **スキップ** | **「新しいメモとして取り込む」を明示的に選べる**（選んだ項目だけ insert） |

- **Phase 1 は insert-only**：どちらの重複でも既存メモの更新・上書きは行わない。**サイレント上書きの経路はコード上存在しない。**
- 「既存メモの明示的な上書き更新モード」は将来の別フェーズ候補（§22）。Phase 1 のスコープ外。
- 既知の限界：取り込み時に新IDを採番するため、**同じファイルをもう一度取り込むと①（id 一致）では検知できない**。この場合は②（タイトル＋本文の完全一致）が防波堤になる。本文を少しでも編集した再取り込みは重複として検知されない（新しいメモとして追加される）＝許容する仕様として明記する。

---

## 11. 不正ファイル・サイズ上限

以下は `invalid`（理由付きスキップ）とし、**他のファイルの処理は止めない**：

- 拡張子が `.md` でない（大文字小文字問わず判定）。
- **1MB（1,048,576 バイト）超**。`DRIVE_MARKDOWN_READ_MAX_BYTES` と同じ値を、`lib/import/` の独立定数（例：`IMPORT_MAX_FILE_BYTES`）として定義する。
- 読み取り不能（FileReader 失敗・壊れたファイル）。
- 解析後にタイトル・本文の両方が空。

すべてのスキップ理由は、プレビューと結果サマリにやさしい日本語で表示する（例：「1MBを超えているため取り込めません」「メモの内容が空のため取り込めません」）。

---

## 12. プレビュー画面（必須）

ファイル選択後、**保存の前に必ず**プレビューを表示する。ファイル選択だけでは何も保存されない。

表示内容：

- 合計サマリ：「取り込み可能 N件／重複 M件／エラー K件」。
- 各ファイル（1行〜1カード）：
  - 状態バッジ：「取り込み可能」「重複（同じメモID）」「重複の可能性」「エラー」。
  - 確定タイトル（フォールバック適用後）・タグ・作成/更新日時（保持かフォールバックかが分かる表示。例：「作成 2025-04-01（元のメモの日時）」「作成 2026-07-18（取り込み時刻）」）。
  - 本文の冒頭抜粋（**約200字**。既存の AI 参照抜粋と同じ長さ基準）。
  - スキップ・エラーの理由。
  - 「重複の可能性」の項目のみ：「新しいメモとして取り込む」チェック（既定 OFF）。
- 注意書き（固定表示）：「既存のメモは変更されません。取り込みは新しいメモの追加だけを行います。」

---

## 13. 保存前の明示的な確認

- プレビュー下部の実行ボタン：「**N件をMyBrainに取り込む**」（N＝取り込み対象件数。0件なら無効化）。
- 押下で**確認ダイアログ**（件数入り）を表示。キャンセルなら何も保存しない。
- **10件以上**のときは、既存の一括エクスポートと同じ段階的警告を先に出す（しきい値を共有）。
- 確認 OK のときだけ保存を開始する。**自動保存・バックグラウンド保存は一切ない。**

---

## 14. 保存先の挙動（Supabase のみ）

- 取り込みの書き込み先は**常に Supabase（`memos` テーブル）だけ**。
- メモの保存先設定（`mybrain` / `obsidian-local` / `obsidian-gdrive`）が**どれであっても挙動は同じ**。保存先設定の読み書き・意味は変更しない。
- **ローカル Vault（Obsidian）への .md 書き戻しはしない。Google Drive への書き出しもしない。**（取り込むファイルは多くの場合そこから来たものであり、書き戻すと重複 .md を作る循環になるため。）
- 取り込み後に改めてエクスポートしたい場合は、既存の手動エクスポート導線を使う（新しい導線は足さない）。

---

## 15. 保存フロー（insert-only）

1. 確認 OK 後、対象（`ok` 全件＋`importAsNew` が ON の `duplicate-content`）を**1件ずつ順に insert** する。
2. insert は §8 の取り込み専用関数で行い、**新しいメモIDを DB が採番**する。`user_id` はログイン中ユーザー（既存 `createMemo` と同じ・RLS 準拠）。
3. タイトル・本文の trim、空タイトル→「無題」は既存の保存挙動をそのまま継承する。
4. **既存メモへの update・delete は一切呼ばない**（insert-only）。
5. 1件の失敗で全体を止めない（§16）。
6. 完了後、メモ一覧を再読み込みして結果サマリを表示する。

---

## 16. 部分失敗の結果報告

- Supabase の insert はブラウザから1行ずつ行うため、**トランザクション・ロールバックは存在しない**。これを隠さず、結果をそのまま報告する。
- 結果サマリ（既存 `GoogleDriveBatchWriteResult` と同じ考え方）：
  - 追加した件数（成功）。
  - スキップした件数（重複・エラー。理由別）。
  - 失敗した件数（insert 失敗。ファイル名＋理由の一覧）。
- **途中失敗時の扱い**：成功済みのメモはそのまま残る（通常のメモとして削除も編集もできる）。「一部のメモは追加済みです。失敗した分だけやり直せます」の旨を表示する。自動リトライ・自動巻き戻しはしない。

---

## 17. プライバシー・セキュリティ・RLS 境界

- 選択したファイルは**ブラウザ内でのみ**読み取り・解析する。**外部サービスへの送信はしない**（Supabase への保存だけが唯一の書き込み）。
- insert は `user_id`＝ログイン中ユーザーで行い、**RLS により自分の行しか作成・閲覧できない**（既存 `createMemo` と同じ境界）。
- 未知の frontmatter キーは読み込まない（旧 Expo 版 CSV インポートの「許可した列以外は無視＝APIキー等の秘密情報は読み込まない」方針を踏襲）。
- ファイル内容を localStorage・Supabase 以外の場所へ保存しない。プレビューは React state のみ（画面を離れれば消える）。
- ログイン必須画面にのみ導線を置く（未ログインでは到達できない）。

---

## 18. デスクトップ先行の UI フロー

- **Phase 1 はデスクトップのみ**。導線は設定画面の既存プレースホルダ「⬇ インポート（データ取込）」（`components/DesktopSettings.tsx`）を接続する。
- 画面フロー：設定 → 「⬇ インポート（データ取込）」 → 取り込みパネル（ファイル選択 → プレビュー → 確認 → 結果サマリ）。パネルの実装位置（モーダルか「データ管理」タブ内セクションか）は IMP2 で最終決定する（推奨：既存タブ構造を活かして「データ管理」タブ内）。
- **スマホ UI（最優先で保護）には一切手を触れない**。モバイル対応は IMP5 で別途扱う（`<input type="file">` は iOS Safari でも動作するため技術的な障害は小さい）。
- デスクトップ先行は、Drive 連携（OBS26→OBS36）で実証済みの「デスクトップで実証 → モバイルへ慎重に展開」の順序を踏襲する。

## 20. QA チェックリスト草案（IMP3 で実施・今回は実施しない）

- **I1**：設定の「⬇ インポート（データ取込）」から取り込み画面が開く（「準備中です」トーストが出ない）。
- **I2**：ファイル選択ダイアログで `.md` を複数選択できる。
- **I3**：21件以上選択すると案内が出て、取り込みが開始されない（最大20件/回）。
- **I4**：frontmatter 付き Markdown が解析され、プレビューにタイトル・タグ・日時・本文抜粋（約200字）が表示される。
- **I5**：frontmatter 無し Markdown は全文が本文になり、タイトルはファイル名（.md 除去）になる。
- **I6**：有効な `created` / `updated` が保持される（取り込み後のメモの日時が frontmatter と一致する）。
- **I7**：`created` 無効・欠落→取り込み時刻／`updated` 無効・欠落→保持した created（無ければ取り込み時刻）／`updated` < `created` →created が使われる。
- **I8**：既存メモと同じ frontmatter `id`＋`source: "mybrain"` のファイルは「重複（同じメモID）」となり、既定でスキップされる。
- **I9**：タイトル＋本文が既存メモと完全一致のファイルは「重複の可能性」となり、既定でスキップされる。
- **I10**：「重複の可能性」の項目だけ「新しいメモとして取り込む」を選べ、選んだ項目だけが新しいメモとして追加される（既存メモは無変更のまま）。
- **I11**：不正ファイル（.md 以外・1MB 超・空・読めない）は理由付きでスキップされ、他のファイルの取り込みは継続する。
- **I12**：ファイル選択だけでは何も保存されない（プレビューを経ずに保存されない）。
- **I13**：「N件をMyBrainに取り込む」→確認ダイアログ OK の後にだけ保存される。キャンセルでは1件も保存されない。10件以上では先に警告が出る。
- **I14**：取り込まれたメモは新しいメモIDを持ち（frontmatter の `id` と異なる）、既存メモは1件も変更・上書きされない。
- **I15**：結果サマリに追加・スキップ・失敗の件数と理由が表示される。
- **I16**：途中で失敗しても成功済みのメモは通常のメモとして残り、その旨が表示される（ロールバックしない・自動リトライしない）。
- **I17**：保存先設定（mybrain / obsidian-local / obsidian-gdrive）のどれでも、取り込みは Supabase のみに保存され、ローカル Vault・Google Drive に .md ファイルは作成されない。
- **I18**：モバイル UI は変化しない（Phase 1 はデスクトップのみ）。
- **I19**：`npx tsc --noEmit`・`npm run build` が成功する。

## 21. 非目標（Phase 1 でやらないこと）

- 自動保存・自動取り込み・バックグラウンド監視。
- 既存メモの上書き・更新・削除（明示的な上書き更新モードも Phase 1 では作らない）。
- 双方向同期。
- Supabase スキーマ変更（IMP0〜IMP3 を通じて不要。`created_at`／`updated_at` は既存カラムを使う）。
- 画像の復元（Markdown は画像データを持たない。`images` は常に `[]`）。
- ZIP／JSON／CSV の取り込み（§4 のとおり後回し・CSV は不採用の方向）。
- モバイル UI の変更（IMP5 まで無変更）。
- OAuth スコープ・Google Drive／Obsidian の挙動・カレンダー連携の変更。
- 取り込み履歴の永続化（どのファイルをいつ取り込んだかの記録は保存しない）。

## 22. 未解決の論点・将来拡張

- **Drive 直接取り込み**：エクスポート済み一覧（`DriveExportedFilesList`）に「このメモを取り込む」を足せば、ファイルダウンロード無しで Drive → MyBrain の取り込みが可能。Phase 1 のパイプライン（判定・プレビュー・確認・insert）をそのまま再利用できる見込み。別 OBS として設計する。
- **JSON エクスポート＋インポート対**：画像を含む完全バックアップの唯一の経路。エクスポート形式の設計が先（設定の「⬆ エクスポート」行と同時に検討）。
- **明示的な上書き更新モード**：「重複（同じメモID）」の項目を、ユーザーが1件ずつ明示的に選んだ場合のみ既存メモを更新する将来オプション。既定は今後もスキップ。要・個別設計（サイレント上書き禁止の原則は維持）。
- **20件/回の上限の見直し**：実運用で不足するなら、結果報告とパフォーマンスを確認した上で緩和を検討。
- **プレビュー中の並行変更**：プレビュー表示中に別タブでメモが増減した場合、重複判定は選択時点のスナップショットに基づく（保存直前の再判定はしない）。実害が出るなら「保存直前に再判定」を検討。
- **再取り込みの検知強化**：§10 の既知の限界（新ID採番により id 検知が2回目以降効かない）への対策として、「取り込み由来メモに元 id をタグや本文フッターで残す」案があるが、メモ本文を汚すため Phase 1 では採用しない。必要になったら再検討。
