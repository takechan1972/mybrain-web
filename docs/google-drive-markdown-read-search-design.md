## 機能名

**Google Drive Markdown 読み取り・検索参照**（Google Drive Markdown Read / Search Reference）

MyBrain が Google Drive の `MyBrain/Memos/` に書き出した Markdown ファイルを、あとから一覧・読み取りし、将来的に AI アシスト・検索の「参照データ」として使えるようにする。

## 目的

- これまでのフェーズで「MyBrain → Drive」への一方向エクスポートが完成した（OBS24R で全9 QAケース Pass）。次は「書き出した Markdown を読み返す」最小の入口を用意する。
- 読み取りは**参照のみ**。MyBrain（Supabase）は引き続き source of truth であり、Drive の Markdown は付加的なエクスポートコピーのまま変えない。

---

## 大前提：スコープと見える範囲（最重要の設計判断）

現在の OAuth スコープは `https://www.googleapis.com/auth/drive.file`（`lib/google/google-drive-oauth.ts` の `GOOGLE_DRIVE_SCOPE`）。

- `drive.file` は「**このアプリが作成（またはユーザーがこのアプリで開いた）ファイルだけ**」を一覧・読み取りできる。
- つまり **MyBrain 自身がエクスポートした Markdown は、スコープ変更なしで読み返せる**。
- 逆に、Obsidian や他アプリ・手動アップロードで `MyBrain/Memos/` に置かれたファイルは**見えない**。これは不具合ではなく `drive.file` の仕様であり、本設計では**許容する制限**とする。
- Drive 全体を読む `drive.readonly` への拡張は、同意画面の重み・審査・プライバシー面の影響が大きいため**採用しない**（スコープ変更禁止のルールどおり）。

この判断により、OBS25 は「MyBrain が自分で書き出したものを自分で読み返す」機能に限定される。

## 設計

### 1. Markdown ファイルの一覧取得

- 既存の `ensureDriveFolderPath`（`lib/google/google-drive-folders.ts`）で `MyBrain/Memos/` のフォルダ ID を解決する（無ければ「まだエクスポートがありません」扱い。読み取りのためにフォルダを新規作成しない）。
- `files.list`（既存 `findDriveFile` と同じエンドポイント・同じ認可ヘッダ）で、フォルダ直下の `.md` ファイルを取得する。
  - クエリ：`'<folderId>' in parents and mimeType!='application/vnd.google-apps.folder' and trashed=false`
  - fields：`files(id, name, modifiedTime, size)`、`orderBy=modifiedTime desc`、`pageSize` は 100 程度＋`nextPageToken` 対応。
- トークンは既存の `requestGoogleDriveAccessToken()` をユーザー操作起点で取得し、**引数で持ち回るだけ・保存しない**（既存方針のまま）。
- 新ヘルパー案：`listDriveMarkdownFiles(accessToken, folderId)` を `lib/google/google-drive-read.ts`（新規）に置く。

### 2. 選択した Markdown 本文の読み取り

- `files.get`：`GET https://www.googleapis.com/drive/v3/files/<fileId>?alt=media` で本文テキストを取得する。
- 一覧から**ユーザーが選んだ 1 件だけ**読み取る（全件の自動一括読み込みはしない）。
- サイズ上限（例：1MB）を超えるファイルは読み込まず、「大きすぎるため表示できません」と案内する。
- 新ヘルパー案：`readDriveMarkdownFile(accessToken, fileId)`（同じく `google-drive-read.ts`）。

### 3. Markdown → メモ相当構造への変換

- **新規パーサは作らない。** 既存の `markdownToMemo(markdown): ParsedMemoMarkdown`（`lib/markdown/memo-markdown.ts`）をそのまま再利用する。
  - frontmatter（`id` / `title` / `tags` / `created` / `updated` / `source: mybrain`）＋本文を復元できる。エクスポートと同じモジュールなので往復の整合が保証される。
- frontmatter が無い・壊れている場合も `markdownToMemo` は安全に空値へフォールバックするため、ファイル名をタイトル代わりに表示する。
- 読み取った内容は **React state（メモリ）のみ**で保持する。Supabase・localStorage・IndexedDB には保存しない（Google カレンダー読み取り表示と同じ方針）。

### 4. AI 検索／AI アシストへの将来接続

- 既存の AI アシスト（`askOllamaConsult`）は、メモ配列から参照コンテキスト（最大20件・各200字）を組み立てて渡す構造になっている。
- Drive から読み取った `ParsedMemoMarkdown` はメモ相当の形（title / tags / body / 日時）なので、**将来はこの配列に「Drive 参照メモ」として追加で渡すだけ**で接続できる。AI 側のロジック変更は不要の見込み。
- 検索も同様：メモ一覧の既存クライアント検索（部分一致）と同じ関数を、読み込み済みの Drive メモに適用するだけにする。
- ベクトル検索・埋め込み・サーバ側インデックスは**このフェーズでは扱わない**（将来の別設計）。

### 5. スコープ外（このフェーズでやらないこと）

- Drive → MyBrain（Supabase）への**取り込み・インポート**（メモとして保存し直すこと）。
- 双方向同期・競合解決・Drive 側の編集の反映。
- 自動・定期・バックグラウンドでの読み込み（読み取りは常にユーザー操作起点）。
- OAuth スコープの変更（`drive.readonly` 等への拡張はしない）。
- 他アプリが置いたファイルの読み取り（`drive.file` では不可視。制限として明記する）。
- 埋め込み・ベクトル検索・全文インデックスの構築。
- Google カレンダー連携・Supabase スキーマ・メモ入力 UI の変更。

### 6. プライバシー・安全の境界

- アクセストークンは**短命・メモリのみ・保存しない**（既存の書き出しと同じ）。リフレッシュトークンは扱わない。
- 読み取った Markdown 本文は**画面表示と AI 参照のためのメモリ保持のみ**。Supabase・localStorage 等へ書き込まない。
- 読み取り専用：この機能から Drive のファイルを**変更・削除・移動しない**。
- AI アシストへ渡すのは、ユーザーが明示的に読み込んだファイルの内容だけ（黙って全ファイルを AI に送らない）。
- AI 送信先は既存の Ollama 設定に従う（この設計で送信先は増やさない）。
- 未ログインではメモ画面自体に到達できない（既存のログインガードのまま）。

### 7. Google Drive スコープの前提

- 使用スコープ：`drive.file`（現状のまま・変更しない）。
- `files.list` / `files.get?alt=media` は `drive.file` の範囲内で、**アプリ自身が作成したファイルに対して**利用できる。
- 追加の同意は不要（同じスコープなので、既存の同意ポップアップと同じ体験）。
- 制限の明記：MyBrain がエクスポートしたファイルだけが一覧に出る。Obsidian 等が置いたファイルは表示されない旨を UI の説明文（簡単な日本語）で伝える。

## Phase 3 詳細設計（OBS28）：Drive Markdown を検索・AIの参照データにする

> **ステータス：設計のみ（未実装・2026-07-09）。** MyBrain（Supabase）が source of truth のまま、明示的に読み込んだ Drive Markdown を「Google Drive参照」として検索・AI アシストで使えるようにする設計。取り込み（インポート）機能ではない。

### 1. 参照メモの読み込み方（ユーザーの明示操作のみ）

- 既存の「エクスポート済み一覧」（`components/DriveExportedFilesList.tsx`）の各ファイルに **「参照に追加」ボタン**を追加する。
- 押したときだけ：既存の `readDriveMarkdownFile` で本文を読み → 既存の `markdownToMemo` で解析 → 「Drive参照メモ」としてメモリに追加する（Phase 2 と同じ読み取り経路・新しい Drive ロジックは作らない）。
- 追加済みファイルは重複追加しない（Drive の fileId で判定）。ボタンは「参照中」表示に変わり、**「参照を解除」**で1件ずつ外せる。全解除ボタンも置く。
- 参照メモの保持は **React state のみ**。ページを離れる／再読み込みで消える（保存しない）。この性質を UI に明記する：「参照はこの画面を開いている間だけ有効です。」
- サイズ上限（1MB）・同意キャンセル・エラー時の挙動は Phase 2 と同じ。

### 2. UI 上のラベル（本体メモとの区別）

- 参照メモには必ず **「Google Drive参照」バッジ**を付ける（一目で分かる単純な日本語ラベル）。
- 参照メモの見た目は本体メモと明確に変える（例：点線枠や薄い背景＋バッジ）。
- 参照メモには編集・削除・お気に入り・フォルダ・エクスポートなどの本体メモ用の操作を**一切出さない**（タップ/クリックで読み取り専用プレビューを開くだけ）。

### 3. 検索への含め方（Phase 3a・最初に実装する範囲）

- 場所：デスクトップのメモ一覧（`DesktopMemos.tsx`）。検索は既存の `visible` フィルタ（title / body / tags の部分一致）と同じ規則を使う。
- **本体メモの一覧・件数・フォルダには参照メモを混ぜない。** 検索語があるときだけ、本体の検索結果の下に独立セクション「Google Drive参照の検索結果（N件）」を表示する。

## Phase 3b 詳細設計（OBS30）：Drive 参照メモを AI アシストのコンテキストに渡す

> **ステータス：設計のみ（未実装・2026-07-10）。** Phase 3a（検索参照・OBS29／OBS29R）が本番QAを通ったので、次の最小ステップとして「ユーザーが明示的に読み込んだ Drive 参照メモを、デスクトップ AI アシストの参照コンテキストに、出所を明示して渡す」設計を確定する。取り込み（インポート）ではない。実装は別の独立タスクとする。

### 0. 前提（現状のコードの確認）

- デスクトップの AI アシストは `components/DesktopMemos.tsx` の `askAssistant(question)`（自由質問）で、既存の `ollamaChat([system, user], settings)` を直接呼ぶ。現在のコンテキストは「選択中メモのタイトル＋本文」だけ（`【メモのタイトル】… 【メモの本文】…`）。
- Drive 参照メモは同じ `DesktopMemos` 内の `driveRefMemos`（`DriveReferenceMemo[]`・メモリのみ・保存しない）で保持されている。`askAssistant` と同じコンポーネントなので、**新しい state も新しい Drive ロジックも増やさず**、この配列を参照ブロックとして user メッセージ末尾に足すだけで接続できる。
- モバイルの AI アシスト（`app/consult` ＋ `lib/ai/consult-ollama.ts` の `askOllamaConsult` / `buildContext`）は**別経路**であり、Phase 3b では**一切変更しない**（モバイル UI 不変のルール）。参照メモの読み込み UI 自体がデスクトップのみなので、自然にデスクトップ限定になる。

### 1. 参照メモを AI コンテキストに含める方法（最小の安全な形）

- 対象は `askAssistant`（デスクトップの自由質問）のみ。要約・整理（`runMemoAi`）は対象外（メモ1件の加工なので参照は足さない）。
- 送信直前に、`driveRefMemos` から**参照ブロック文字列**を組み立て、既存の user メッセージの**後ろに別ブロックとして追記**する。既存の `【メモのタイトル】…【メモの本文】…` の構造は変えない。
- 参照が0件のときは**何も足さない**（既存の送信内容と完全に同じ＝挙動不変）。
- `ollamaChat` の呼び出し形・引数・送信先は変えない。`lib/ai/consult-ollama.ts` も変えない（デスクトップは別経路のため）。
- 組み立ては `DesktopMemos` 内の小さなヘルパー（例：`buildDriveReferenceBlock(refs)`）で完結させる想定。

### 2. 出所を明示するラベル（本体メモと混ぜない見出し）

- 参照ブロックには必ず出所を明示する見出しを付ける。例：

  ```
  【Google Drive参照メモ（MyBrain本体ではありません／エクスポート済みファイルの参考）】
  1. タイトル：<title または ファイル名>
     本文（抜粋）：<body を1行化して先頭N字>
  2. …
  ```

- system プロンプトにも1文だけ足す（案）：「『Google Drive参照メモ』は補助的な参考情報です。回答の主な根拠は選択中のメモ（MyBrain本体）とし、参照メモを使ったときはその旨が分かるように答えてください。」
- 本体メモの `【メモの本文】` に参照本文を混ぜ込まない（必ず別見出しの別ブロックにする）。

### 3. 上限（件数・文字数）

- **参照メモの最大件数：5件**（`DRIVE_REF_AI_MAX_ITEMS = 5`）。読み込み済みが5件を超える場合は、リストの先頭から5件だけを使う（超過分は送らない）。
- **1件あたりの本文最大文字数：200字**（`DRIVE_REF_AI_MAX_CHARS = 200`）。既存 `buildContext`（メモ200字）と同じ基準にそろえる。超過は先頭200字＋「…」。
- タイトルも長い場合は適度に切る（例：60字）。改行・連続空白は1スペースに正規化してから切る（既存 `buildContext` と同じ整形）。
- 上限は「AI に渡しすぎない・遅くしない・混乱させない」ための安全弁。将来変えるときは1か所（定数）で調整できるようにする。

### 4. 使うのは「今読み込んでいる参照」だけか

- **はい。** AI に渡すのは、そのとき `driveRefMemos` に入っている（＝ユーザーがこの画面で「参照に追加」した）ものだけ。
- 自動読み込み・過去分の復元・Drive 再問い合わせは**しない**。参照を「すべて解除」した後は0件として扱い、何も渡さない。
- 検索語で参照ブロックを絞るかは**あえてしない（全件・上限内で渡す）**を採用する：ユーザーが自分で選んで追加した少数の参照なので、そのまま渡す方が 10〜70代にも予測しやすい。将来必要なら「検索語に一致する参照だけ渡す」に狭める余地は残す（この設計では広い側の全件を既定にする）。

### 5. 本体メモ（source of truth）と黙って混ぜないための原則（must）

- 参照メモは常に**別見出し・別ブロック**で渡す。本体メモの本文に連結しない。
- 件数・お気に入り・フォルダ・一覧・一括選択・一括エクスポートには引き続き一切混ぜない（Phase 3a の原則を維持）。
- AI が参照を使うときは、UI 側の事前表示（次項）とプロンプト内の見出しの両方で「Drive参照も参考にした」ことが分かるようにする。黙って全ファイルを送らない（ユーザーが追加した分だけ）。
- Supabase には保存も問い合わせもしない。参照はメモリのみ。

### 6. UI 通知テキスト（送信前に分かる形）

- 参照が1件以上あるとき、AI アシスタント入力の近く（送信ボタンの上など）に小さな一文を出す：

  > **Google Drive参照 N件も参考にします**（MyBrainには保存されません）

- 実際に上限で切られた場合は件数を実送信数に合わせる（例：7件読み込み済みなら「Google Drive参照 5件も参考にします」）。
- 回答生成中・完了後も、参照を使ったことが分かるよう同じ表示を残す（黙って混ぜない）。
- 文言は簡単な日本語のみ（10〜70代向け）。専門用語（トークン・コンテキスト等）は UI に出さない。

### 7. スコープ外（Phase 3b でやらないこと）
