# Google Drive Markdown 書き出し連携 ― 設計メモ

> **ステータス：設計のみ（未実装）。** このドキュメントは実装前の方針整理であり、アプリコード・OAuth・Drive API 呼び出しは含まない。
> 関連：`docs/obsidian-vault-export-design.md`（ローカル Vault 書き出し）／`lib/markdown/export-memos-zip.ts`／`lib/fs/`（ローカル FS コア）。

---

## 機能名

**Google Drive Markdown 書き出し連携**（Google Drive Markdown Export）

選択したメモを、ユーザーの Google Drive 上の `MyBrain/Memos/` に Obsidian 互換 Markdown（`.md`）として書き出す。

---

## 目的

- ローカル Vault 直接書き出しは Chromium 系デスクトップに限られる。**モバイルや非対応ブラウザでも**クラウド経由で Vault 互換ファイルを置けるようにする。
- 端末を問わず、メモを Obsidian 互換 Markdown として持ち出せる経路を増やす（「ハブアプリ」構想の一部）。
- Supabase は引き続き source of truth。Drive 書き出しは**コピーの一方向エクスポート**。

## なぜ Google Drive が有用か

- 主要 OS・ブラウザ・モバイルから広く使える（File System Access API の制約を受けない）。
- Obsidian の Vault を Google Drive 上に置いて同期しているユーザーが多く、**Drive に置けばそのまま Vault に反映**されるケースがある。
- 端末間共有・バックアップを Google 側に委ねられる。

## ローカル Vault 書き出しとの違い

| 観点 | ローカル Vault（File System Access） | Google Drive |
|---|---|---|
| 対応環境 | Chromium 系デスクトップのみ | デスクトップ／モバイル広く |
| 認証 | 不要（フォルダ選択のみ） | OAuth が必要 |
| 保存先参照 | `FileSystemDirectoryHandle`（IndexedDB 保存） | Drive のフォルダ ID（＋トークン） |
| ネットワーク | 不要（ローカル書き込み） | 必須（API 経由アップロード） |
| 既存ファイル判定 | ディレクトリ内のファイル存在チェック | Drive 検索（フォルダ ID ＋名前） |

両者の**コア（Markdown 化・重複名回避・件数確認・安全ルール）は共通化**し、書き出し先だけをアダプタとして差し替える。

---

## MVP スコープ

1. ユーザーが Google アカウントで連携（OAuth 同意）する導線。
2. `MyBrain/Memos/` フォルダを Drive 上に確保（無ければ作成）。
3. 選択メモを既存の `createMemoMarkdownFile(memo)` で Markdown 化（生成ロジックは再利用・複製しない）。
4. アップロード前に**件数を含む確認ダイアログ**を出す。
5. 同名ファイルがある場合は**上書きせず**連番（`名前-2.md`）で新規作成。
6. 結果（成功／失敗の件数、失敗タイトル）をトースト等で通知（ローカル書き出しと同じ方針）。
7. デスクトップ・モバイル双方で利用可能。

## スコープ外（このフェーズでやらないこと）

- 双方向同期（Drive → MyBrain の取り込み・競合解決）。
- 自動・定期アップロード（バックグラウンド同期）。
- Drive 上のメモ編集の読み戻し。
- 複数アカウント・複数フォルダの同時管理。
- Google Calendar / Sheets 連携（別機能）。
- Supabase スキーマ／保存挙動の変更。

---

## OAuth 概要（このユースケースで使う範囲）

- Google OAuth 2.0 で**ユーザー同意**を得てアクセストークンを取得する。
- Web アプリ向けのフロー（Authorization Code ＋ PKCE、または Google Identity Services のトークンフロー）を想定。
- **ユーザー操作起点**でのみ同意フローを開始（勝手にログインしない）。
- リフレッシュトークンを使う場合はサーバー側保管が前提（下記トークン方針）。MVP は最小権限・短命トークン中心で検討。

## 必要な Google API スコープ

- 第一候補：**`drive.file`**（このアプリが作成・オープンしたファイルのみアクセス）。
  - 最小権限。MyBrain が作った `MyBrain/Memos/` 配下だけ扱え、ユーザーの他のファイルには触れない。
- 広い `drive`（全ファイルアクセス）は**使わない**（過剰権限）。
- 既存 Vault フォルダへ正確に書き込む必要が出た場合のみ、`drive.file` の範囲で「ユーザーがピッカーで選んだフォルダ」を対象にする方式（Google Picker）を検討。

## トークン保管方針

- **保存する場合も最小限**。理想は**サーバー側（バックエンド）でのみリフレッシュトークンを保持**し、クライアントには短命のアクセストークンのみ渡す。
- クライアント（ブラウザ）には**リフレッシュトークンを保存しない**。アクセストークンもメモリ保持を基本とし、長期保存は避ける。
- localStorage への生トークン保存は**しない**（XSS 露出リスク）。
- 失効・取り消しに耐えるよう、トークン無効時は再同意フローに戻す。

### 保存するもの

- 連携状態（接続済みかどうか）の最小フラグ。
- 書き出し先フォルダの Drive フォルダ ID（再利用のため）。
- 任意で表示用のアカウント識別（メールの一部など、最小限）。

### 保存しないもの

- メモ本文・Markdown の中身（Supabase が source of truth）。
- Supabase のデータ。
- リフレッシュトークン（クライアント側には保存しない）。
- 個人情報（必要最小限を超える Google プロフィール情報）。

## Drive フォルダ構造（案）

ローカル Vault と**同じ構造**に揃える：

```
（Drive のルート or ユーザー指定フォルダ）/
└─ MyBrain/
   └─ Memos/
      ├─ 買い物メモ.md
      ├─ 会議メモ.md
      └─ 無題のメモ.md
```

- パス／ファイル名は既存の `createMemoMarkdownFile`（`MyBrain/Memos`、ファイル名規則）に揃え、ローカル書き出し・ZIP と結果一致。
- Drive ではフォルダは ID で扱うため、`MyBrain` → `Memos` の各フォルダ ID を解決／作成して保持する。

## Markdown 書き出しの挙動

1. 選択メモを `createMemoMarkdownFile` で `{ fileName, content }` 化。
2. `MyBrain/Memos/` フォルダ ID を確保（無ければ作成）。
3. ファイルごとに：
   - 既存チェック（フォルダ ID ＋ファイル名で Drive 検索）。
   - 無ければ作成してアップロード。
   - あれば**上書きせず**連番（`名前-2.md`）で新規作成。
4. 進捗・結果（成功／失敗の件数、失敗タイトル）を集計して通知。
5. 失敗があっても処理は止めず、最後にまとめて報告（ローカル書き出しと同方針）。

### 一方向書き出しのみ

- 本フェーズは **MyBrain → Drive の一方向**に限定。Drive 側の変更は読み戻さない。

### アップロード前のユーザー確認

- 複数ファイルをアップロードする前に、**必ず件数つき確認ダイアログ**を出す（`MyBrain/Memos/` 明記）。
- 大量選択時は警告（しきい値はローカルと揃え 10 件）。

### 黙ってアップロードしない

- ユーザー操作（クリック）起点でのみアップロードを実行。自動・定期アップロードはしない。

### 既存 Drive ファイルを黙って上書きしない

- 同名ファイルは上書きせず連番で新規作成（下記）。

### 重複ファイル名の扱い

- フォルダ内に同名がある場合、`名前-2.md` `名前-3.md` … と連番を付ける（ローカル・ZIP と同じ規則）。

## エラーハンドリング方針

| 事象 | 対応 |
|---|---|
| 未連携／同意していない | アップロード導線を無効化し、連携を案内 |
| OAuth 同意キャンセル | 何もしない（無音で終了） |
| トークン失効・取り消し | 再同意フローへ誘導し、処理は中断 |
| ネットワーク／API エラー | その1件を失敗計上し継続、最後に件数報告 |
| レート制限（429 等） | リトライ間隔を空ける／件数が多い場合は警告。無理に連投しない |
| フォルダ作成失敗 | 中止し、失敗を通知 |
| 想定外の例外 | 失敗トースト（既存文言に揃える） |

## モバイル・デスクトップの挙動

- **デスクトップ**：ローカル Vault 直接書き出し（実装済み）と Drive 書き出しの両方を提供しうる。
- **モバイル**：ローカル直接書き出しは非対応のため、**Drive 書き出しがクラウド経路の主役**になる（ZIP との併用）。
- どちらも OAuth 同意・件数確認・上書き回避の安全ルールは共通。

---

## Obsidian との関係

- Drive 上の `MyBrain/Memos/` は **Obsidian 互換 Markdown**（既存の YAML frontmatter ＋本文）で保存。
- Obsidian の Vault を Google Drive 同期している場合、Drive に置けば Vault に取り込まれうる。
- ローカル Vault 書き出しと**ファイル構造・命名を一致**させ、経路が違っても同じ結果にする。

## Google Calendar との関係

- 本機能は **Drive へのファイル書き出し**であり、Calendar 連携（予定の取得・登録）とは**別機能・別スコープ**。
- 認証基盤（Google OAuth）を将来共有しうるが、スコープは機能ごとに最小化する（Drive と Calendar のスコープを混ぜない）。

## Google Sheets との関係

- Sheets 連携（表形式エクスポート等）も**別機能・別スコープ**。
- Markdown 書き出しの対象外。共通化するのは OAuth 基盤の考え方までで、データ形式・保存先は別。

---

## 将来の双方向同期について（注記のみ）

- 本フェーズおよび Drive 書き出しは**一方向（MyBrain → Drive）**に限定。
- 双方向同期（Drive の編集を MyBrain へ取り込む）は、競合解決・更新時刻比較・削除の扱いなど別途設計が必要で、**現時点ではスコープ外**。
- 双方向に進む場合も、Supabase を source of truth とする前提を崩さない設計から検討する。
